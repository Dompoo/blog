**** Q. “사용자 관점에서 문제를 정의했다”고 하셨는데, 실제 사례를 들어 설명해 주세요.**

예를 들어, MAU가 증가함에도 평균 세션 시간이 그대로인 상황이 있었는데요. 왜 세션 시간이 그대로인지 확인하려고 노력했어요. 서비스를 직접 사용해보며 원하는 코스를 찾기까지의 시간을 측정해보았고, 1분 보다 훨씬 많은 시간이 소요된다는 것을 발견했습니다.
따라서 저희는 이를 중간에 이탈한 것으로 결론내리고, UT를 통해서 이탈한 이유를 찾기 위해 노력한 결과 코스의 정확도가 낮은 것이 주된 문제라는 것을 알게되었습니다.

**Q. UT에서 나온 인사이트가 이전 가설과 다르게 흘러간 경험이 있나요? 그때 어떻게 판단을 바꾸셨나요?**

각 코스에 대하여 총 길이, 도로 타입을 표시해주고 있었는데, 다음으로 더 필요한 메타데이터를 '난이도'라고 생각하고 있었어요. 따라서 최고 난이도를 10으로, 최저 난이도를 0으로 모델링하는 기능을 개발하고, 이를 화면에서 표현해주고 있었는데요.
UT를 해보니까 0~10의 표현보다는 쉬움 보통 어려움 등의 한국어로 표현하는 것이 더 잘 와닿는다는 이야기를 들을 수 있었어요. 따라서 0부터3을 쉬움, 3부터7을 보통, 7부터10을 어려움으로 책정하고 표현 방식을 바꿨던 경험이 있습니다.

**Q. “빠른 실패와 피드백”이 실제로 성과로 이어진 구체적 사례를 설명해 주세요.**

저희 프로젝트는 도메인 특성상 GPX 파일이 매우 중요한데요. 이 GPX 파일을 관리하기 위하여 파일 스토리지가 필요했습니다. S3 버킷이나 다른 서비스들을 고려했었는데, 해당 기능이 실제로 어떻게 변경될지 몰랐기 때문에 안드로이드 개발자도 익숙하고 SDK 덕분에 개발하기 편했던 google drive를 빠르게 선택했습니다.
추후 프로젝트가 점점 진화함에 따라 해당 GPX 파일 스토리지가 필요없어졌습니다. 따라서 빠르고 쉽게 구현했던 기능을 큰 손실없이 제거할 수 있었고, 시간 단축을 할 수 있었습니다.

**Q. 유저 보이스 3건을 기반으로 길찾기 기능을 만들었다고 했는데, 데이터 수가 적은 상황에서 어떻게 의사결정을 검증하셨나요?**

일단 실제 사용자가 없는 상황에서 최대한 데이터를 많이 쌓아야 통계적으로 올바른 유저 보이스를 들을 수 있을 것이라고 생각했어요.
따라서 실제 UT와 설문을 병행했습니다. 설문은 우아한테크코스 내부의 이전 선배들이나 코치님들도 모두 볼 수 있는 게시판에 업로드했고, 100분 정도가 응답해주셨습니다.
이를 통해서 사용자가 없는 상황에서도 최대한 통계적으로 올바른 의사결정을 내리려고 노력했습니다.

**Q. 사용자 행동 변화(길찾기 기능 사용률 증가)를 어떻게 측정했고, 어떤 지표를 가장 중요하게 봤나요?**

해당 부분은 서버 로그를 통해 측정했습니다. 저희 팀은 API 호출에 대해 로그를 남기고 있었는데요. 따라서 특정 구간의 주변 코스 조회 API 호출 수와 길찾기 API 호출 수를 계산해낼 수 있었습니다.
길찾기 API 호출 수를 주변 코스 조회 API 호출 수로 나누면 사용자가 검색한 코스에 비해 얼마나 많은 길찾기를 했는지 알 수 있기 때문에 비교적 정확한 계산이라고 판단하여 진행했습니다.
